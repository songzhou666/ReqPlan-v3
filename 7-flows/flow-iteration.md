# 需求迭代流程 (Requirement Iteration Flow)

## 流程概述
对现有项目进行功能迭代、bug修复或性能优化的增量开发流程。
**Harness 适配**：L / M / H — L 级别走简化 3 阶段管道（入口 → 执行 → 回写），M/H 走完整 5 阶段。

## 适用场景
- 现有功能优化
- 新功能添加
- Bug修复
- 性能改进
- 用户体验优化

## 流程步骤

### 步骤1：迭代需求分析
**目标**：分析本次迭代的需求和目标
**引导**：
```
🔄 需求迭代 - 分析阶段

请描述本次迭代的目标：
1. 要解决什么问题？
2. 要添加什么功能？
3. 有什么性能要求？

输入 /reqplan intent <迭代描述> 开始分析
```
**输出**：迭代需求文档
**writeback_target**：`docs/req/req-{{taskId}}.md`
**完成检查**：迭代目标明确、需求文档创建

### 步骤2：影响分析
**目标**：分析变更对现有系统的影响
**引导**：
```
🔍 影响分析阶段

分析以下方面的影响：
- 现有功能影响
- 数据库变更影响
- API接口变更影响
- 测试用例影响

输入 /reqplan sync 进行影响分析
```
**输出**：影响分析报告
**writeback_target**：`docs/harness/project-constraints.md`
**完成检查**：影响范围明确、风险已识别

### 步骤3：迭代计划
**目标**：制定本次迭代的开发计划
**引导**：
```
📅 迭代计划阶段

将创建迭代计划：
- 迭代任务分解
- 时间安排
- 验收标准
- 风险应对

输入 /reqplan plan 制定迭代计划
```
**输出**：迭代计划文档
**writeback_target**：`.agent/plans/plan-iteration-{{taskId}}.md`
**完成检查**：计划文档创建、任务已分解

### 步骤4：代码开发
**目标**：执行迭代开发任务
**引导**：
```
💻 迭代开发阶段

当前迭代任务：
1. [ ] TASK-IT-001 - 修改数据库结构
2. [ ] TASK-IT-002 - 实现新功能
3. [ ] TASK-IT-003 - 更新接口文档

输入 /reqplan task list 查看所有任务
```
**输出**：代码变更、任务进度
**writeback_target**：`PR/MR`
**完成检查**：所有迭代任务完成

### 步骤5：测试验证
**目标**：验证迭代功能和回归测试
**引导**：
```
🧪 迭代测试阶段

执行以下测试：
- 新功能测试
- 回归测试
- 性能测试
- 兼容性测试

输入 /reqplan verify 执行验证
```
**输出**：测试报告
**writeback_target**：`docs/test/verify-{{taskId}}.md`
**完成检查**：所有测试通过

### 步骤6：文档更新
**目标**：更新相关文档
**引导**：
```
📚 文档更新阶段

需要更新的文档：
- 需求文档
- 设计文档
- API文档
- 用户手册

输入 /reqplan docs 管理文档更新
```
**输出**：更新后的文档
**writeback_target**：`docs/（各文档目录）`
**完成检查**：相关文档已更新

### 步骤7：迭代验收
**目标**：验收迭代成果
**引导**：
```
✅ 迭代验收阶段

验收检查清单：
✅ 新功能正常工作
✅ 回归测试通过
✅ 文档已更新
✅ 无已知严重bug

输入 /reqplan verify 完成验收
```
**输出**：验收结果
**writeback_target**：`任务系统（更新状态）`
**完成检查**：迭代验收通过

## 流程进度
```
步骤进度：1/7 完成
总体进度：14%
```

## 分支选项
- 步骤2后：可跳转到设计评审流程（如涉及设计变更）
- 步骤4后：可跳转到代码审计流程
- 步骤5后：可跳转到测试优化流程

## 完成标准
- 迭代需求文档完成
- 影响分析清晰完整
- 迭代任务全部完成
- 相关测试通过
- 文档已更新
- 迭代验收通过

## 失败策略
- 发现重大问题时可暂停迭代
- 支持回滚到迭代前状态
- 提供重新规划选项

## 适用边界

### 适合场景
- 已有项目的增量开发、功能迭代、Bug修复周期

### 轻量处理（可直接自动化）
- 纯文本修改、注释修复

### 需人工介入
- 优先级变更、范围蔓延判断
