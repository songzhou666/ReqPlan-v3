  - 用户说"出错了"、"报错了"、"修个Bug"、"修复..."
  - 用户说"帮我规划..."、"怎么做..."、"有什么方案"

  2. 按状态机执行：START→ANALYZE→CONFIRM→DESIGN→IMPLEMENT→VERIFY→JUDGE

  5. 用户必须在 CONFIRM 阶段确认后才能继续

  - 验证报告（_verification.md）

# ReqPlan-v3 — Harness Engineering 引擎

> **核心设计原则**：约束即自由。给 AI 严格的框架，它才能在框架内交出可靠的工作。
>
> **强制执行原则**：本 Skill 的每个规则都必须通过"检查点"验证，未通过则阻断流程。

---

## ⚠️ 激活后的强制入口检查（不做就无法继续）

### 入口检查清单（必须逐项确认）

```markdown
## 强制入口检查（激活后立即执行）

### 步骤 0：确认项目路径（防止读错接力棒）
- [ ] 0.1 我已确认当前项目路径：{具体路径}
- [ ] 0.2 如果用户未指定路径 → 我已询问："请提供项目路径"
- [ ] 0.3 如果用户说"继续昨天的" → 我已询问："请提供昨天的项目路径"
- [ ] 0.4 我确认不会读取其他项目的接力棒

### 步骤 1：停止当前行为
- [ ] 1.1 我已停止任何编码/分析/设计行为
- [ ] 1.2 我没有输出任何代码、设计或分析内容
- [ ] 1.3 我当前的回复只包含"正在初始化..."或类似提示

### 步骤 2：读取接力棒（必须执行具体命令）
- [ ] 2.1 我已执行：read {项目路径}/.agent/harness/_baton.md
- [ ] 2.2 如果文件存在 → 我已看到文件内容
- [ ] 2.3 如果文件不存在 → 我已执行：mkdir -p {项目路径}/.agent/harness/
- [ ] 2.4 如果文件不存在 → 我已创建接力棒并写入状态 START

### 步骤 3：确定状态
- [ ] 3.1 我已从接力棒中提取当前状态
- [ ] 3.2 我已确定当前状态是：{具体状态名}
- [ ] 3.3 如果无法确定状态 → 我已将状态设为 START

### 步骤 4：确定下一步
- [ ] 4.1 我已根据当前状态确定下一步操作
- [ ] 4.2 当前状态为 START → 下一步：进入 ANALYZE
- [ ] 4.3 当前状态为 ANALYZE → 下一步：继续分析或展示产物
- [ ] 4.4 当前状态为 CONFIRM → 下一步：展示摘要等待用户
- [ ] 4.5 当前状态为 DESIGN → 下一步：读取分析产物开始设计
- [ ] 4.6 当前状态为 IMPLEMENT → 下一步：读取设计产物开始实现
- [ ] 4.7 当前状态为 VERIFY → 下一步：读取设计产物开始验证
- [ ] 4.8 当前状态为 JUDGE → 下一步：读取验证报告做判定

**阻断规则**：以上任一未勾选 → 我必须停止 → 先完成入口检查 → 才能继续

**如果我没有接力棒**：
→ 立即执行：mkdir -p {项目路径}/.agent/harness/
→ 立即创建：{项目路径}/.agent/harness/_baton.md
→ 写入内容：状态 START，模式 NORMAL，重试计数 0
→ 继续执行

**如果我忘记执行入口检查**：
→ 用户会看到我直接开始编码/分析
→ 这是错误行为 → 我必须立即停止 → 重新执行入口检查

**验证方式**：
→ 我必须在回复中明确写出："当前状态：[状态名]，下一步：[操作]"
→ 如果我没有写出这句话 → 说明我没有执行入口检查 → 必须重新执行
```

### 入口检查后的强制路由

| 当前状态 | 我必须做的事 | 我禁止做的事 |
|----------|-------------|-------------|
| START | 创建接力棒，进入 ANALYZE | ❌ 直接开始编码 |
| ANALYZE | 读取 analyzer-agent.md，生成 _analysis.md | ❌ 跳过分析直接设计 |
| CONFIRM | 展示摘要，等待用户响应 | ❌ 自动进入下一阶段 |
| DESIGN | 读取 _analysis.md 和 designer-agent.md | ❌ 不读分析就设计 |
| IMPLEMENT | 读取 _design.md 和 implementer-agent.md | ❌ 不读设计就编码 |
| VERIFY | 读取 _design.md 和 verifier-agent.md | ❌ 不读设计就验证 |
| JUDGE | 读取 _verification.md，做判定 | ❌ 不看验证报告就做判断 |
| DONE/ABORT/FAILED | 输出最终报告，流程结束 | ❌ 继续执行 |

## 二、阶段转换的强制检查点

### 2.1 检查点机制（Checkpoint）

每个阶段结束前，必须通过检查点。检查点未通过 → 阻断流程 → 必须修复后才能继续。

```
阶段执行流程：
┌─────────────────┐
│ 1. 读取接力棒    │ ← 强制
└────────┬────────┘
         ▼
┌─────────────────┐
│ 2. 读取阶段指南  │ ← 强制（如 analyzer-agent.md）
└────────┬────────┘
         ▼
┌─────────────────┐
│ 3. 执行阶段任务  │
└────────┬────────┘
         ▼
┌─────────────────┐
│ 4. 生成产物      │
└────────┬────────┘
         ▼
┌─────────────────┐
│ 5. 自检清单      │ ← 强制
└────────┬────────┘
         ▼
┌─────────────────┐
│ 6. 检查点验证    │ ← 强制阻断
└────────┬────────┘
         ▼
┌─────────────────┐
│ 7. 更新接力棒    │ ← 强制
└────────┬────────┘
         ▼
┌─────────────────┐
│ 8. 进入下一阶段  │
└─────────────────┘
```

### 2.2 各阶段检查点清单

#### ANALYZE → CONFIRM 检查点

```markdown
## 检查点：ANALYZE 完成验证

### 产物检查（全部通过才能继续）
- [ ] _analysis.md 文件已创建
- [ ] 文件包含 "# 需求分析报告" 标题
- [ ] 文件包含 "## 基本信息" 章节
- [ ] 文件包含 "## 需求理解" 章节
- [ ] 文件包含 "## 技术栈" 章节
- [ ] 文件包含 "## 涉及文件" 章节
- [ ] 文件包含 "## 约束条件" 章节

### 阻断条件
如果以上任一检查项未通过：
→ 停止流程
→ 输出错误："ANALYZE 阶段产物不完整，缺少：[具体缺失项]"
→ 要求重新执行 ANALYZE 阶段
```

#### CONFIRM → DESIGN 检查点

```markdown
## 检查点：CONFIRM 完成验证

### ⚠️ 强制等待机制（防止 AI 自动跳过）

**阻断规则**：在收到用户明确回复前，禁止执行任何其他操作。

**禁止行为**：
- ❌ 不要自动进入下一阶段
- ❌ 不要假设用户同意
- ❌ 不要基于"推测"继续执行
- ❌ 不要在等待期间开始编码或设计

### 用户确认检查（全部通过才能继续）
- [ ] 用户明确回复"确认"、"同意"、"继续"、"是的"等肯定词
- [ ] 或用户未提出修改意见

### 如果用户未回复或回复不明确
→ 停止所有操作
→ 再次展示需求摘要
→ 明确提示用户回复【确认/修改/取消】
→ 继续等待

### 阻断条件
如果用户选择"修改"：
→ 返回 ANALYZE 阶段
→ 记录修改内容到接力棒

如果用户选择"取消"：
→ 状态设为 ABORT
→ 输出终止报告
→ 流程结束

如果用户未明确确认：
→ 停止流程
→ 输出提示："请确认需求理解是否正确（确认/修改/取消）"
```

#### DESIGN → IMPLEMENT 检查点

```markdown
## 检查点：DESIGN 完成验证

### 产物检查
- [ ] _design.md 文件已创建
- [ ] 文件包含 "# 技术设计文档" 标题
- [ ] 文件包含 "## 技术方案概述" 章节
- [ ] 文件包含 "## 模块划分" 章节
- [ ] 文件包含 "## 任务列表" 章节（带依赖关系）
- [ ] 文件包含 "## 验证方案" 章节（Layer 1-5）

### 可执行性检查
- [ ] 任务列表中的每个任务都有明确的"涉及文件"
- [ ] 任务列表中的每个任务都有明确的"验证方式"

### 阻断条件
如果以上任一检查项未通过：
→ 停止流程
→ 输出错误："DESIGN 阶段产物不完整或不可执行，缺少：[具体缺失项]"
```

#### IMPLEMENT → VERIFY 检查点

```markdown
## 检查点：IMPLEMENT 完成验证

### 产物检查
- [ ] _implementation.md 文件已创建
- [ ] 文件包含 "# 实现摘要" 标题
- [ ] 文件包含 "## 完成的任务" 章节
- [ ] 文件包含 "## 涉及的文件" 章节

### 代码检查
- [ ] 所有设计中的任务都已完成（或明确标记为未完成）
- [ ] 代码已保存到文件系统（不是只存在于对话中）

### 阻断条件
如果代码未保存到文件：
→ 停止流程
→ 输出错误："代码必须保存到文件后才能进入验证阶段"
```

#### VERIFY → JUDGE 检查点

```markdown
## 检查点：VERIFY 完成验证

### 产物检查
- [ ] _verification.md 文件已创建
- [ ] 文件包含 "# 验证报告" 标题
- [ ] 文件包含 Layer 1-5 的验证结果
- [ ] 文件包含 "## 综合判定" 章节（PASS/FAIL）
- [ ] 如果 FAIL，包含错误分类（ARCHITECTURE/REVIEW/RUNTIME/ENVIRONMENT）

### 阻断条件
如果判定结果不明确：
→ 停止流程
→ 输出错误："VERIFY 阶段必须给出明确的 PASS/FAIL 判定"
```

## 三、防跳过机制

### 3.1 阶段跳跃阻断

**禁止行为**：从当前状态直接跳到非相邻状态。

```
如果当前状态是 ANALYZE：
  ❌ 禁止直接跳到 IMPLEMENT
  ❌ 禁止直接跳到 VERIFY
  ✅ 只能进入 CONFIRM

如果当前状态是 DESIGN：
  ❌ 禁止直接跳到 VERIFY（必须先 IMPLEMENT）
  ✅ 只能进入 IMPLEMENT
```

**阻断输出**：
```
错误：检测到阶段跳跃。
当前状态：[当前状态]
目标状态：[目标状态]
正确路径：[当前状态] → [正确下一状态]
请按正确路径执行。
```

### 3.2 产物缺失阻断

**禁止行为**：不读取前置产物就执行当前阶段。

```
进入 DESIGN 前：
  必须读取 _analysis.md
  如果 _analysis.md 不存在 → 阻断

进入 IMPLEMENT 前：
  必须读取 _design.md
  如果 _design.md 不存在 → 阻断

进入 VERIFY 前：
  必须读取 _design.md 和 _implementation.md
  如果任一不存在 → 阻断

进入 JUDGE 前：
  必须读取 _verification.md
  如果 _verification.md 不存在 → 阻断
```

## 四、防遗忘机制

### 4.1 接力棒强制更新模板

每个阶段结束后，必须按以下模板更新接力棒。不更新 → 视为阶段未完成。

```markdown
## 元信息
- 当前状态: {状态名称}
- 模式: {NORMAL/DESIGN_FIX/REVIEW_FIX/RETRY_FIX}
- 重试计数: {0/1/2}
- design_fix_retry: {0/1/2}
- 最后更新: {ISO 8601}

## 进度追踪
- [x] START - {时间}
- [x] ANALYZE - {时间} ✅
- [x] CONFIRM - {时间} ✅
- [ ] DESIGN - 进行中 ⏳
- [ ] IMPLEMENT - 待开始
- [ ] VERIFY - 待开始
- [ ] JUDGE - 待开始

## 产物清单
- [x] _analysis.md ✅
- [x] _design.md ✅
- [ ] _implementation.md ⏳
- [ ] _verification.md ⏳

## 当前阶段详情
### {状态名称}（进行中）
**进度**: {百分比}%
**已完成**: {列表}
**进行中**: {任务}
**待完成**: {列表}

## 下一步行动
1. {具体任务1}
2. {具体任务2}
```

### 4.2 跨 Session 续跑指南

**场景**：用户昨天开始任务，今天继续

**续跑步骤**：
1. 询问用户项目路径（如果不知道）
2. 执行：read {项目路径}/.agent/harness/_baton.md
3. 从接力棒提取：
   - 当前状态
   - 已完成阶段
   - 进行中的任务
   - 待完成的任务
4. 在回复中明确写出："检测到续跑请求，当前状态：[状态]，继续执行..."
5. 按当前状态继续执行

**如果用户说"继续"但没有指定项目**：
→ 询问："请提供项目路径，以便读取接力棒恢复进度"

### 4.3 状态遗忘自检

每次用户输入后，AI 必须自问：

```markdown
## 状态自检（每次交互前必须执行）

1. 当前状态是什么？（从接力棒读取）
2. 上一阶段是否已完成？（检查产物是否存在）
3. 当前阶段应该做什么？（读取阶段指南）
4. 是否需要用户确认？（CONFIRM 阶段检查）

如果无法回答以上任何问题：
→ 读取接力棒
→ 如果接力棒不存在 → 创建接力棒
→ 重新确定状态
```

## 六、错误处理与阻断

### 6.1 常见错误阻断

| 错误场景 | 阻断输出 | 修复指引 |
|----------|----------|----------|
| 接力棒不存在 | "错误：接力棒不存在。请先执行 /reqplan init" | 创建接力棒 |

| 产物缺失 | "错误：前置产物 [文件名] 不存在，无法进入 [阶段]" | 重新执行前置阶段 |
