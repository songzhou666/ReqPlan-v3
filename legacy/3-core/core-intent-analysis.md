# Core Intent Analysis
# 意图分析模块

## 概述

意图分析是ReqPlan的核心能力之一，负责理解用户输入，识别流程意图，并为后续的文档生成和任务管理提供基础分析。

## 核心原则

1. **模糊容忍**：允许用户输入不完整、不精确的需求描述，通过追问逐步明确
2. **渐进披露**：不一次性要求所有信息，而是在对话中分步引导用户提供
3. **意图分类优先**：先确定用户想让系统做什么（触发哪个流程），再处理具体内容
4. **上下文保留**：每次分析结果都记录到状态文件中，支持多轮对话

## Input/Output定义

### Input

```yaml
- description: string  # 用户自然语言描述
- context?: object     # 项目上下文（可选）
```

**字段说明**：
- description：用户描述的需求，可以是完整的句子或关键词组合
- context：可选的上下文信息，包括项目名称、技术栈、已有需求等

### Output

```yaml
- functionalPoints: string[]    # 功能点列表
- userRoles: string[]          # 用户角色列表
- dataFlow: object            # 数据流程
- summary: string             # 意图摘要
```

**字段说明**：
- functionalPoints：从需求中提取的所有功能点
- userRoles：涉及的用户角色及其权限
- dataFlow：数据的来源、去向和处理流程
- summary：对整个需求的简要总结

## 详细流程

### Step 1: 理解核心功能点

**任务**：
1. 识别用户需求中的核心动词（CRUD操作）
2. 识别涉及的业务对象
3. 列出所有识别到的功能清单

**输出**：
```markdown
识别到的功能点：
1. [功能1]
2. [功能2]
3. [功能3]
```

**询问**：
```markdown
请确认是否遗漏了其他功能？
- [ ] 需要
- [ ] 不需要
```

### Step 2: 识别用户角色

**任务**：
1. 识别涉及的用户角色
2. 明确每个角色的权限
3. 确定角色间的关系

**输出**：
```markdown
用户角色：
1. [角色1] - 权限：[权限列表]
2. [角色2] - 权限：[权限列表]
```

**常见角色**：
- 管理员：具有所有权限
- 普通用户：基础CRUD权限
- 审核员：只读权限
- 客户：受限的CRUD权限

### Step 3: 识别数据来源

**任务**：
1. 确定数据来源（用户录入、系统生成、外部接口）
2. 确定数据存储位置（数据库、缓存、文件）
3. 确定数据处理流程

**输出**：
```markdown
数据流程：
- 来源：[数据来源]
- 存储：[存储位置]
- 处理：[处理流程]
```

### Step 4: 确认关键信息

**任务**：
1. 询问补充信息（如需要）
2. 提供明确的选项让用户选择
3. 确认理解是否正确

**常见补充问题**：
```markdown
为更好理解需求，请确认：

1. 优先级：
   - P0（紧急）
   - P1（重要）
   - P2（一般）

2. 技术约束：
   - 性能要求
   - 安全要求
   - 兼容性要求

3. 交付时间：
   - 紧急
   - 常规
   - 灵活
```

### Step 5: 生成意图分析

**任务**：
1. 输出结构化分析结果
2. 保存到 intent.md
3. 提供下一步建议

**输出格式**：
```markdown
# 意图分析报告

## 基本信息
- 需求ID：INTENT-001
- 创建时间：[时间]
- 状态：已确认

## 功能点
[功能点列表]

## 用户角色
[用户角色及权限]

## 数据流程
[数据流程说明]

## 优先级
[P0/P1/P2]

## 技术约束
[约束条件]

## 摘要
[简要总结]
```

## 失败策略

### E001: 信息不足

**错误信息**：
```yaml
E001: 信息不足，无法理解需求
```

**处理方式**：
```markdown
请补充以下信息：

1. [缺失的信息1]
2. [缺失的信息2]

请选择或直接描述：
```

**示例**：
```
用户：我需要一个订单功能
AI：  请补充以下信息：
      1. 核心功能是什么？（创建订单、查询订单、取消订单）
      2. 谁会使用这个功能？（管理员、客户、运营）
      3. 数据从哪里来？（用户录入、API导入）
```

### E002: 需求模糊

**错误信息**：
```yaml
E002: 需求过于模糊，需要澄清
```

**处理方式**：
```markdown
我理解您需要[理解的需求]，但需要澄清以下几点：

1. [需要澄清的问题1]
2. [需要澄清的问题2]

请提供更多信息：
```

**示例**：
```
用户：做一个好的系统
AI：  我理解您需要"做一个好的系统"，但需要澄清以下几点：
      1. 具体是哪种系统？（电商、CRM、OA、库存管理等）
      2. 主要功能是什么？
      3. 使用对象是谁？
```

## 示例对话

### 示例1：完整需求分析

```
用户：我需要一个订单管理模块，包括订单创建、查询、修改和取消功能
AI：  好的，我来帮你分析这个需求。

      **识别到的功能点**：
      1. 订单创建
      2. 订单查询
      3. 订单修改
      4. 订单取消

      请确认：
      1. 是否遗漏了其他功能？
      2. 用户角色是什么？（管理员、订单处理员、客户）

用户：还要加上订单统计功能，用户角色是管理员
AI：  明白了！

      **识别到的功能点**：
      1. 订单创建
      2. 订单查询
      3. 订单修改
      4. 订单取消
      5. 订单统计

      **用户角色**：管理员（所有权限）

      还需要确认：
      1. 数据来源是什么？（用户录入、API导入）
      2. 有没有性能要求？（响应时间、并发量）
      3. 优先级是什么？（P0/P1/P2）

用户：数据从用户录入，没有特别的性能要求，优先级P1
AI：  ✅ 意图分析完成

      **意图摘要**：
      - 功能点：5个（CRUD + 统计）
      - 用户角色：管理员
      - 数据来源：用户录入
      - 优先级：P1

      已生成意图分析文档：INTENT-001.md

      下一步建议：
      1. 生成完整需求文档
      2. 进行API设计
      3. 制定开发计划
```

## 最佳实践

1. **先理解再确认**：不要急于生成文档，先确保理解正确
2. **提供明确选项**：减少用户的输入成本
3. **分步确认**：逐步确认每个关键点
4. **及时生成文档**：确认后立即生成结构化文档
5. **反馈闭环**：信息不足时及时返回上一步
