# Core Review
# 审核建议模块

## 概述

审核建议是ReqPlan的核心能力之一，负责提供代码和文档的一致性审核、质量分析和改进建议。
**边界**：review 负责人工/半自动判断（设计一致性、代码可读性、架构合理性），输出分级报告和评分；自动化工具检查（lint/typecheck/test）由 core-verification.md 处理。

## 核心原则

1. **全面覆盖**：从一致性、质量、规范、安全、性能五个维度综合评估
2. **分级报告**：问题按严重程度（Critical/Major/Minor/Info）分级呈现，优先处理高严重性问题
3. **建议可执行**：每个问题附带具体的修复建议，而非模糊的批评
4. **量化评估**：质量评分机制让改进有据可依，便于追踪代码质量趋势

## Input/Output定义

### Input

```yaml
- target: "code" | "doc" | "integration"
- content: string  # 代码或文档内容
- context?: object # 上下文信息（可选）
```

**字段说明**：
- target：审核目标类型
  - code：代码审核
  - doc：文档审核
  - integration：集成审核
- content：需要审核的代码或文档内容
- context：可选的上下文信息

### Output

```yaml
- issues: Issue[]       # 发现的问题列表
- suggestions: string[] # 改进建议
- score: number        # 质量评分
```

**字段说明**：
- issues：发现的问题列表
- suggestions：改进建议列表
- score：质量评分（0-100）

## 审核类型

### 1. 代码审核（code）

**审核维度**：
```yaml
- 一致性：代码与需求文档的一致性
- 质量：代码的可读性、可维护性
- 规范：命名规范、注释规范、格式规范
- 安全：潜在的安全漏洞
- 性能：性能问题识别
```

### 2. 文档审核（doc）

**审核维度**：
```yaml
- 完整性：是否包含所有必要章节
- 准确性：内容是否准确无误
- 一致性：文档之间的一致性
- 可读性：文档结构是否清晰
- 时效性：文档是否过期
```

### 3. 集成审核（integration）

**审核维度**：
```yaml
- 接口一致性：API设计与实现的一致性
- 数据一致性：前后端数据模型的一致性
- 流程一致性：业务流程的一致性
```

## Issue对象

```yaml
- id: string              # 问题ID（ISSUE-001）
- severity: "critical" | "major" | "minor" | "info" # 严重程度
- type: string           # 问题类型
- location: string      # 问题位置
- description: string     # 问题描述
- impact: string       # 影响说明
```

### 严重程度定义

```yaml
critical: 必须立即修复，否则会导致系统无法运行
major: 应该尽快修复，影响系统功能
minor: 建议修复，影响代码质量
info: 信息性建议，不影响功能
```

## 详细流程

### Step 1: 读取审核目标

**任务**：
1. 识别审核目标类型
2. 加载相关文档
3. 收集上下文信息

**代码审核**：
```markdown
正在读取代码...
已加载：
- 代码文件：X个
- 代码行数：Y行
- 关联需求：REQ-001
```

**文档审核**：
```markdown
正在读取文档...
已加载：
- 文档：design.md
- 关联文档：req.md, plan.md
```

### Step 2: 执行审核

**代码审核检查项**：

```markdown
- [ ] 一致性检查
      - 代码是否与需求一致？
      - 接口签名是否正确？
      
- [ ] 代码质量检查
      - 圈复杂度是否过高？
      - 函数长度是否合适？
      - 是否有重复代码？
      
- [ ] 规范检查
      - 命名是否规范？
      - 注释是否充分？
      - 格式是否统一？
      
- [ ] 安全检查
      - 是否有SQL注入风险？
      - 是否有XSS风险？
      - 敏感信息是否泄露？
      
- [ ] 性能检查
      - 是否有性能问题？
      - 数据库查询是否优化？
```

### Step 3: 生成报告

**任务**：
1. 整理发现的问题
2. 生成改进建议
3. 计算质量评分

**报告格式**：
```markdown
# 审核报告

## 基本信息
- 审核类型：{{target}}
- 审核时间：{{timestamp}}
- 质量评分：{{score}}/100

## 发现问题

### 🔴 Critical（必须修复）
- ISSUE-001: [问题描述]
  - 位置：[代码位置]
  - 影响：[影响说明]
  - 建议：[修复建议]

### 🟠 Major（应该修复）
- ISSUE-002: [问题描述]
  ...

### 🟡 Minor（建议修复）
- ISSUE-003: [问题描述]
  ...

## 改进建议

1. [建议1]
2. [建议2]
3. [建议3]

## 总结

总体评价：[评价]
建议行动：[下一步行动]
```

## 失败策略

### E301: 审核目标无效

**错误信息**：
```yaml
E301: 审核目标无效：{{target}}
```

**处理方式**：
```markdown
⚠️ 审核目标无效：{{target}}

支持的审核类型：
- code：代码审核
- doc：文档审核
- integration：集成审核

请重新指定审核目标。
```

### E302: 内容为空

**错误信息**：
```yaml
E302: 审核内容为空
```

**处理方式**：
```markdown
⚠️ 审核内容为空

请提供需要审核的代码或文档内容。
```

## 示例对话

### 示例1：代码审核

```
用户：帮我审核一下这段代码
AI：  好的，正在进行代码审核...

      📊 代码审核报告

      **基本信息**：
      - 代码行数：150行
      - 函数数量：5个
      - 审核时间：2026-05-14

      **质量评分**：78/100

      **发现的问题**：

      🔴 ISSUE-001 [Critical]
      - 问题：SQL注入风险
      - 位置：userRepository.js:23
      - 影响：可能导致数据库被攻击
      - 建议：使用参数化查询

      🟠 ISSUE-002 [Major]
      - 问题：函数过长
      - 位置：orderService.js:45-90
      - 影响：代码难以维护
      - 建议：拆分为多个小函数

      🟡 ISSUE-003 [Minor]
      - 问题：缺少注释
      - 位置：payment.js:15
      - 影响：代码可读性差
      - 建议：添加函数说明注释

      **改进建议**：

      1. 立即修复SQL注入问题（Critical）
      2. 重构过长的函数（Major）
      3. 添加必要的注释（Minor）

      **总结**：
      代码整体结构良好，但存在1个严重问题需要立即修复。
      建议优先修复Critical级别的问题。
```

## 最佳实践

1. **全面检查**：覆盖所有审核维度
2. **分级处理**：按严重程度分类问题
3. **具体建议**：每个问题都有明确的修复建议
4. **量化评分**：给出质量评分便于对比
5. **持续改进**：记录问题避免重复出现
