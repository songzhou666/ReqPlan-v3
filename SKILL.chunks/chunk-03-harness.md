# 验证与审查体系

来源：Harness Engineering 分层验证模型

验证不只是在最终阶段做一次，而是嵌入到流程的每个阶段。

## 五层验证金字塔

```
                 ┌────────────────────────┐
                 │  Layer 5: 回写验证      │
                 │  链路记录是否完整？      │
                 ├────────────────────────┤
                 │  Layer 4: 失败验证      │
                 │  异常、回滚策略是否符合？ │
                 ├────────────────────────┤
                 │  Layer 3: 链路验证      │
                 │  入口→处理→输出是否连通？ │
                 ├────────────────────────┤
                 │  Layer 2: 单元验证      │
                 │  核心函数、边界是否正确？ │
                 ├────────────────────────┤
                 │  Layer 1: 静态检查      │
                 │  代码风格、类型、格式？  │
                 └────────────────────────┘
```

### 验证失败处理

按层逐级执行，当前层失败则不进入下一层：

```
Layer N 失败 → 记录失败详情 → 修复问题 → 重试 Layer N
              → 如果连续3次重试仍无法通过：
                → 记录已知例外到 plan 文件末尾
                → 询问用户是否跳过该层继续
                → 用户确认后进入下一层
```

所有验证命令由 AI 在项目环境中直接运行。如果命令不存在（如项目未配置 lint），跳过该检查项并记录说明。

### Layer 1：静态检查

| 检查项 | 命令示例 | 通过标准 |
|--------|---------|---------|
| 代码风格 | npm run lint / ruff check / go vet | 零错误或仅允许的 warning |
| 类型检查 | npm run typecheck / mypy / tsc --noEmit | 零类型错误 |
| 格式检查 | prettier --check / black --check / gofmt -l | 无格式差异 |

### Layer 2：单元验证

| 检查项 | 通过标准 |
|--------|---------|
| 核心函数正确性 | 关键 case 全部通过 |
| 边界条件 | 空值、极值、异常输入已覆盖 |
| 错误路径 | try-catch 分支已覆盖 |

### Layer 3：链路验证

| 检查项 | 说明 |
|--------|------|
| 入口连通 | 命令/API 能否正常调用 |
| 输入输出 | 数据能否正确流转 |
| 外部依赖 | 数据库/API/文件系统是否正常连通 |

### Layer 4：失败验证

| 检查项 | 说明 |
|--------|------|
| 失败重试 | 超时、网络异常时的重试行为是否符合预期 |
| 回滚策略 | 计划中定义的 Rollback 是否可执行 |
| 降级行为 | 依赖不可用时是否有合理的降级 |

### Layer 5：回写验证

| 检查项 | 说明 |
|--------|------|
| 链路记录 | history.yaml 是否已追加本任务记录 |
| 决策日志 | 关键选型是否已记入 decisions.yaml |
| 约束登记 | 重复出现的问题是否已登记到 project-constraints.md |

> 回写验证的详细指导见 chunk-04-chain.md

## 审查清单

分层验证通过后，进入人工/半自动审查环节：

### 代码质量

- 命名是否表达意图
- 函数/组件职责单一
- 复杂度是否在合理范围
- 是否有重复代码

### 设计一致性

- 实现与设计方案是否一致
- 接口约定是否被遵守
- 数据模型与设计文档是否对齐

### 安全性

- 用户输入是否有校验
- 权限控制是否到位
- 敏感信息是否泄露

### 异常处理

- 错误路径是否正确处理
- 失败时是否有明确提示
- 日志是否记录了关键信息

## 前端交付三层检查

来源：Harness Engineering Pencil → Storybook → 页面

```
Step 1 执行依据层 → 页面结构/组件层级设计确认
Step 2 状态暴露层 → Default/Empty/Loading/Error/权限态 方案就绪
Step 3 交付实现层 → 路由/API/权限/联调完成
```

---

## 后端交付三层检查

```
Step 1 执行依据层 → 入口/输入/输出/异常口径已说清
Step 2 验证口径层 → 验证命令/mock/复现方案就绪
Step 3 交付实现层 → 实现/测试/联调完成
```
