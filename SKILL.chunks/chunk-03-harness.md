# 验证与审查体系

来源：Harness Engineering 分层验证模型

验证不只是在最终阶段做一次，而是嵌入到流程的每个阶段。

## 五层验证金字塔

```
                 ┌────────────────────────┐
                 │  Layer 5: 流程合规      │
                 │  产物完整性、文档是否齐？ │
                 ├────────────────────────┤
                 │  Layer 4: 异常处理      │
                 │  异常、回滚策略是否符合？ │
                 ├────────────────────────┤
                 │  Layer 3: 构建集成      │
                 │  编译、依赖、构建是否通？ │
                 ├────────────────────────┤
                 │  Layer 2: 单元测试      │
                 │  核心函数、边界是否正确？ │
                 ├────────────────────────┤
                 │  Layer 1: 静态检查      │
                 │  代码风格、类型、格式？  │
                 └────────────────────────┘
```

### 验证失败处理

按层逐级执行，当前层失败则不进入下一层：

```
Layer N 失败 → 记录失败详情 → 修复问题 → 重试 Layer N
              → 如果连续3次重试仍无法通过：
                → 记录已知例外到 plan 文件末尾
                → 询问用户是否跳过该层继续
                → 用户确认后进入下一层
```

所有验证命令由 AI 在项目环境中直接运行。如果命令不存在（如项目未配置 lint），跳过该检查项并记录说明。

### Layer 1：静态检查

| 检查项 | 命令示例 | 通过标准 |
|--------|---------|---------|
| 代码风格 | npm run lint / ruff check / go vet | 零错误或仅允许的 warning |
| 类型检查 | npm run typecheck / mypy / tsc --noEmit | 零类型错误 |
| 格式检查 | prettier --check / black --check / gofmt -l | 无格式差异 |

### Layer 2：单元测试

| 检查项 | 通过标准 |
|--------|---------|
| 核心函数正确性 | 关键 case 全部通过 |
| 边界条件 | 空值、极值、异常输入已覆盖 |
| 错误路径 | try-catch 分支已覆盖 |

### Layer 3：构建集成

| 检查项 | 说明 |
|--------|------|
| 编译检查 | 项目能否正常编译 |
| 依赖安装 | 依赖解析是否正确 |
| 构建输出 | 产物是否按预期生成 |

### Layer 4：异常处理

| 检查项 | 说明 |
|--------|------|
| 失败重试 | 超时、网络异常时的重试行为是否符合预期 |
| 回滚策略 | 计划中定义的 Rollback 是否可执行 |
| 降级行为 | 依赖不可用时是否有合理的降级 |

### Layer 5：流程合规

| 检查项 | 说明 |
|--------|------|
| 链路记录 | history.yaml 是否已追加本任务记录 |
| 决策日志 | 关键选型是否已记入 decisions.yaml |
| 约束登记 | 重复出现的问题是否已登记到 project-constraints.md |

> 回写验证的详细指导见 chunk-04-chain.md

## 审查清单

分层验证通过后，进入人工/半自动审查环节：

### 代码质量

- 命名是否表达意图
- 函数/组件职责单一
- 复杂度是否在合理范围
- 是否有重复代码

### 设计一致性

- 实现与设计方案是否一致
- 接口约定是否被遵守
- 数据模型与设计文档是否对齐

### 安全性

- 用户输入是否有校验
- 权限控制是否到位
- 敏感信息是否泄露

### 异常处理

- 错误路径是否正确处理
- 失败时是否有明确提示
- 日志是否记录了关键信息

## 前端交付三层检查

来源：Harness Engineering Pencil → Storybook → 页面

```
Step 1 执行依据层 → 页面结构/组件层级设计确认
Step 2 状态暴露层 → Default/Empty/Loading/Error/权限态 方案就绪
Step 3 交付实现层 → 路由/API/权限/联调完成
```

---

## 后端交付三层检查

```
Step 1 执行依据层 → 入口/输入/输出/异常口径已说清
Step 2 验证口径层 → 验证命令/mock/复现方案就绪
Step 3 交付实现层 → 实现/测试/联调完成
```

## 📋 独立质量审核机制（新增）

ReqPlan-v3 在 ANALYZE/DESIGN/IMPLEMENT/VERIFY/JUDGE 阶段启用了**独立子Agent盲审**机制。

### 审核关卡分布

```
ANALYZE → [审核① 分析质量] → CONFIRM → DESIGN → [审核② 设计质量] → IMPLEMENT → [审核③ 实现质量] → VERIFY → [审核④ 验证质量] → JUDGE[审核⑤ 全局判定]
```

### 审核的三级判定

| 等级 | 分数 | 含义 | 动作 |
|------|------|------|------|
| A | ≥90分 | 优秀 | 直接通过 |
| B | 75-89分 | 良好 | 有条件通过 |
| C | 60-74分 | 及格 | 打回修复 |
| D | <60分或阻断 | 不合格 | 打回重做 |

### 核心设计

1. **上下文隔离**：审核子Agent通过 `Task(general_purpose_task)` 拉起，不知道原始需求、前置分析细节
2. **报告持久化**：审核结果写入 `_quality_audit_{phase}.md`，含"待修复问题清单"
3. **修复验证闭环**：重审时逐条检查上次问题是否已修复，防止AI"假装修复"
4. **审核等级**：L1快速检查 / L2标准审核 / L3完整审核（按变更范围选择）
