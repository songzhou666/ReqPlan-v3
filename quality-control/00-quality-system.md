# 质量审核体系定义

> 本文档定义了 ReqPlan-v3 Harness 系统的独立质量审核标准。
> 所有审核子Agent(Quality Auditor)以此文档为唯一评分依据。

---

## 一、审核的三级递进判定体系

每个审核关卡执行**三级递进判定**：

```
第1级: 阻断检查（一票否决）
  → 发现任何阻断项 → 直接 FAIL，不再评分
  → 必须引用产物具体内容作为证据

第2级: 维度评分（量化评估）
  → 按 N 个维度分别打分（0-100分）
  → 每个维度有明确的可检查项列表
  → 每个扣分项必须指定扣分理由并引用产物具体内容

第3级: 综合判定（等级评定）
  → A级(≥90分): 优秀，直接通过
  → B级(75-89分): 良好，有条件通过（需标注待改进项）
  → C级(60-74分): 及格，打回修复后重新审核
  → D级(<60分或阻断): 不合格，打回重做
```

## 二、分析阶段审核维度

**审核对象**：_analysis.md

### 维度①：业务影响链路完整性（权重25%）

| 检查项 | 扣分规则 |
|--------|---------|
| 功能变更追溯 | 遗漏1个模块 → 扣15分 |
| 依赖关系图谱 | 无依赖图 → 扣20分 |
| 级联影响分析 | 未做二级以上分析 → 扣15分 |
| 回归范围圈定 | 未圈定 → 扣10分 |

**过关标准**：
- ✅ 过关：依赖链至少追溯到2层以上，明确标注回归范围
- ❌ 不过关：只列了直接涉及的模块，没有做链路追溯

### 维度②：上下游校验完整度（权重20%）

| 检查项 | 扣分规则 |
|--------|---------|
| 上游输入校验 | 遗漏来源 → 扣15分 |
| 下游输出校验 | 遗漏去向 → 扣15分 |
| 数据流转路径 | 路径断点 → 扣10分 |
| 接口契约变化 | 未识别 → 扣20分 |

**过关标准**：
- ✅ 过关：输入来源和输出去向全部识别，接口变化逐项标注
- ❌ 不过关：只描述模块本身功能，没说数据从哪来到哪去

### 维度③：边界兼容与异常兜底（权重25%）

| 检查项 | 扣分规则 |
|--------|---------|
| 边界条件枚举 | 缺边界场景 → 扣10分/个 |
| 兼容性分析 | 未评估 → 扣15分 |
| 异常兜底策略 | 无兜底 → 扣20分 |
| 数据一致性保障 | 未提及 → 扣15分 |

**过关标准**：
- ✅ 过关：边界场景≥3个，有明确的向前兼容方案，有失败兜底策略
- ❌ 不过关：只说"正常流程"，没提边界和异常

### 维度④：风险预判与依赖追溯（权重30%）

| 检查项 | 扣分规则 |
|--------|---------|
| 依赖链完整追溯 | 断链 → 扣20分 |
| 连锁风险预判 | 无预判 → 扣20分 |
| 风险等级界定 | 无分级 → 扣10分 |
| 缓解/回退方案 | 无方案 → 扣15分 |

**过关标准**：
- ✅ 过关：每个高/中风险有缓解方案，回退方案明确可执行
- ❌ 不过关：只列风险不列方案

## 三、设计阶段审核维度

**审核对象**：_design.md

### 维度①：架构一致性（权重30%）

| 检查项 | 扣分规则 |
|--------|---------|
| 架构模式匹配 | 不符 → 扣20分 |
| 分层合理性 | 跨层调用 → 扣15分/处 |
| 依赖方向正确性 | 循环依赖 → **阻断** |
| 模块内聚性 | 不合理聚合 → 扣10分 |

**阻断条件**：
```
⚠️ 循环依赖: moduleA → moduleB → moduleA          → 阻断
⚠️ 反模式: Controller直接操作Repository             → 阻断
⚠️ 职责混淆: 一个类既做数据校验又做日志又做业务计算  → 扣大分
```

### 维度②：模块边界清晰度（权重25%）

| 检查项 | 扣分规则 |
|--------|---------|
| 接口定义完备性 | 缺定义 → 扣10分/个 |
| 模块职责单一 | 职责模糊 → 扣15分 |
| 依赖外部化 | 直接依赖实现 → 扣10分 |
| 扩展性预留 | 未考虑 → 扣5分 |

### 维度③：接口完整定义（权重25%）

| 检查项 | 扣分规则 |
|--------|---------|
| RESTful规范 | 不规范 → 扣10分 |
| 参数校验 | 缺校验 → 扣10分/个 |
| 异常响应定义 | 缺定义 → 扣15分 |
| 数据模型完整 | 缺字段 → 扣5分/个 |

### 维度④：任务可执行性（权重20%）

| 检查项 | 扣分规则 |
|--------|---------|
| 任务粒度合理 | 过度耦合 → 扣10分 |
| 依赖关系明确 | 缺依赖 → 扣10分 |
| 验收标准具体 | 模糊标准 → 扣10分 |
| 涉及文件精确 | 模糊路径 → 扣5分 |

## 四、实现阶段审核维度

**审核对象**：_implementation.md

### 维度①：设计覆盖度（权重30%）

| 检查项 | 扣分规则 |
|--------|---------|
| 任务完整实现 | 遗漏1个任务 → 扣20分 |
| 功能点对应 | 无法对应设计文档 → 扣15分 |
| 设计约束遵守 | 违反约束 → 扣15分 |

**过关标准**：
- ✅ 过关：所有设计文档中的任务在实现摘要中都有对应条目
- ❌ 不过关：有设计任务未在实现摘要中提及或明确标记未实现

### 维度②：代码规范性（权重25%）

| 检查项 | 扣分规则 |
|--------|---------|
| 文件保存完整性 | 代码未保存 → **阻断** |
| 代码风格一致性 | 严重偏离项目风格 → 扣15分 |
| 注释与文档 | 缺少必要注释 → 扣10分 |

**过关标准**：
- ✅ 过关：所有代码文件已保存到磁盘，风格与项目一致
- ❌ 不过关：有代码文件未保存到磁盘

### 维度③：完整性（权重25%）

| 检查项 | 扣分规则 |
|--------|---------|
| 新增文件列全 | 遗漏文件 → 扣20分 |
| 修改文件追溯 | 未列出修改文件 → 扣15分 |
| 依赖变更记录 | 未记录新增依赖 → 扣10分 |

**过关标准**：
- ✅ 过关：所有新增和修改的文件都在实现摘要中完整列出
- ❌ 不过关：有文件变更未在实现摘要中记录

### 维度④：一致性（权重20%）

| 检查项 | 扣分规则 |
|--------|---------|
| 设计与实现一致性 | 实现偏离设计 → 扣20分 |
| 技术栈一致性 | 使用了未约定的技术 → 扣15分 |
| 接口签名匹配 | 接口签名不一致 → 扣15分 |

**过关标准**：
- ✅ 过关：实现内容与设计文档中的约束条件和技术选型一致
- ❌ 不过关：实现使用了设计文档未约定的技术或框架

## 六、JUDGE阶段全局审核维度

**审核对象**：所有产物

| 维度 | 权重 | 评分依据 |
|------|------|---------|
| ① 需求分析质量 | 20% | _analysis.md 的分析深度、影响链路完整度、风险预判质量 |
| ② 架构设计质量 | 20% | _design.md 的架构合理性、模块边界清晰度、接口完备性 |
| ③ 代码实现质量 | 20% | 代码可读性、错误处理、性能、安全、命名 |
| ④ 测试覆盖质量 | 15% | 覆盖率、边界测试、异常测试、独立性 |
| ⑤ 验证报告质量 | 15% | _verification.md 的完整性、判定准确性 |
| ⑥ 流程规范遵守 | 10% | 是否遵循了状态机、是否更新了接力棒、产物是否完整 |

### 全局阻断条件（一票否决）

- 存在任何未被修复的安全漏洞
- 核心业务逻辑存在严重缺陷
- 存在循环依赖或架构反模式
- 存在明文密钥/密码硬编码
- 构建/编译完全失败且无法修复

## 七、审核等级（L1/L2/L3）

| 等级 | 适用场景 | 审核维度 | 审核方式 |
|---------|---------|---------|---------|
| **L1 - 快速检查** | 注释修改、简单参数调整、纯配置变更 | 仅检查阻断条件 | 主Agent自检，不拉起子Agent |
| **L2 - 标准审核** | 单模块功能新增/修改、非核心逻辑变更 | 维度①+④ 或 维度②+③（按需选择2个） | 拉起子Agent，但只审关键维度 |
| **L3 - 完整审核** | 跨模块变更、架构调整、核心逻辑重构、API变更 | 全部4个维度 | 拉起子Agent，完整审核 |

## 八、修复验证闭环规则

### 重审时子Agent的检查逻辑

```
1. 读取上次审核报告，获取"待修复问题清单"
2. 逐一检查产物中的对应位置
3. 对每个问题：
   - 如果确实修复了 → 标记 ✅，记录证据引用
   - 如果没修复或只修了表面 → 标记 ❌，继续扣分
4. 存在任一 ❌ 未修复项 → 判定为"假修复" → 整体不通过
```

### 产物版本号检查

审核子Agent检查项增加：
- [ ] 产物是否包含版本号？
- [ ] 如果是重审轮次，产物版本号是否已递增？
- [ ] 上次审核报告的P0问题是否已全部得到回应？

## 九、审核"假通过"防御机制

### 防御1：评分必须引用产物具体内容（硬性要求）
```
❌ 无效评分："业务影响链路完整，给90分"
✅ 有效评分："业务影响链路追溯了3层依赖（具体见第2章第3节的依赖图），
                标注了回归范围（用户/订单/支付模块），给90分"
```

### 防御2：阻断判定必须证据引用
```
❌ 无效阻断："存在安全风险，阻断"
✅ 有效阻断："第5行代码 user_input = request.args.get('name') 直接拼接SQL，
                存在SQL注入风险，阻断。引用：_analysis.md 第3章第1节"
```

### 防御3：综合评分分布监控
- 如果连续3次审核均给出 ≥90分（A级），触发"评分偏移"告警
- 主Agent需要对比历史审核报告的评分分布
- 发现异常 → 标记该轮审核报告为"可能不可靠"，要求人工确认

## 十、人工介入条件矩阵

| 触发条件 | 当前状态 | 人工需要做什么 |
|---------|---------|--------------|
| 同一阶段重试2次仍不通过 | FAILED | 审查审核报告，判断是标准太严还是产物确实有问题 |
| 审核子Agent连续3次评分≥90（评分偏移怀疑） | 告警（不阻断） | 人工抽检审核质量，确认是否需要调整标准 |
| 阻断条件涉及敏感安全漏洞（如密码硬编码） | 阻断+告警 | 立即审查产物，确认安全风险级别 |
| AI认为需要人工决策的边界问题（如是否兼容某个老旧版本） | 暂停等待 | 给出决策，继续流程 |
| 产物大范围改动（涉及文件>10个或模块>3个） | 建议人工确认 | 在大范围变更前做一次人工checkpoint |

## 十一、审核超时/失败降级方案

```
拉起子Agent → 等待返回
  │
  ├─ ✅ 成功返回结构化结果 → 正常处理
  │
  ├─ ⚠️ 返回格式异常（缺少必要字段）
  │     → 记录错误到接力棒
  │     → 重新拉起（最多重试1次）
  │     → 仍失败 → 降级为"主Agent自检模式"
  │
  └─ ❌ 完全失败（Task调用异常）
        → 记录错误到接力棒
        → 降级为"主Agent自检模式"
        → 在产物中标注"本次审核为自检模式，未启用独立子Agent"
        → 流程继续（不因审核工具故障而阻塞整个项目）
```

**降级自检模式的要求**：
- 主Agent必须按照相同的审核清单逐项检查
- 必须在审核报告中标注"自检模式"和原因
- 自检模式仅限技术故障场景，不可主动选择

## 十二、等级判定汇总

| 等级 | 分数 | 含义 | 动作 |
|------|------|------|------|
| A | ≥90分 | 优秀 | 直接通过，建议可选的微调 |
| B | 75-89分 | 良好 | 有条件通过，记录待改进项到接力棒 |
| C | 60-74分 | 及格 | 打回修复，进入对应阶段的修复模式 |
| D | <60分或阻断 | 不合格 | 打回重做，从对应阶段重新开始 |
| 重试超限 | - | 无法修复 | FAILED，需人工介入 |
