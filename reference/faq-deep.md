## Q1：Windows 和 Unix 环境下的命令差异怎么处理？

**问题**：我在 Windows 上使用 ReqPlan-v3，但示例命令看起来像 Unix/Linux 命令，能用吗？

**解答**：
ReqPlan-v3 的所有命令设计为跨平台兼容的。Windows 环境下使用 PowerShell 替代 Bash：

| 操作 | Linux/Mac | Windows PowerShell |
|------|-----------|-------------------|
| 创建目录 | `mkdir -p` | `New-Item -ItemType Directory -Force` |
| 读取文件 | `cat file.md` | `Get-Content file.md`（`cat` 别名也支持）|
| 写入文件 | `echo "content" > file.md` | `Set-Content -Path file.md -Value "content"` |
| 路径分隔符 | `/` | `\`（也支持 `/`） |
| 路径测试 | `test -f file.md` | `Test-Path file.md` |

> 注意：PowerShell 的 `cat` 默认输出有限行数，大文件建议使用 `Get-Content file.md -Tail 20`。
> 产物验证时使用 `Test-Path` 替代 `test -f`。

## Q2：接力棒文件丢失了怎么办？

**问题**：我不小心删除了 `.agent/harness/_baton.md` 文件，当前任务还能恢复吗？

**解答**：
可以恢复，按以下步骤操作：

1. **确认残留产物**：检查 `.agent/harness/` 目录下还有哪些产物文件
   ```powershell
   Get-ChildItem ".agent/harness/"
   ```

2. **重建接力棒**：根据现有产物推断当前状态
   - 有 `_analysis.md` → 至少到 ANALYZE 阶段
   - 有 `_analysis.md` + `_design.md` → 至少到 DESIGN 阶段
   - 有 `_analysis.md` + `_design.md` + `_implementation.md` → 至少到 IMPLEMENT 阶段

3. **手动创建接力棒**：使用 `protocols/baton-protocol.md` 中的最小化模板，
   设置状态为推断出的阶段

4. **续跑**：重新激活 Skill，AI 会自动读取新的接力棒并按状态机续跑

> 建议：如果对当前状态不确定，保守起见从 ANALYZE 阶段重新开始，
> 利用已有的产物文件加速分析过程。

## Q3：质量审核不通过时，如何查看具体问题？

**问题**：我的 ANALYZE 阶段被 Quality Auditor 打回了，说"审核不通过"，但具体问题在哪里？

**解答**：
读取审核报告文件查看详细问题清单：

```markdown
read {项目路径}/.agent/harness/_quality_audit_{阶段}.md
```

审核报告的结构包括：
- **阻断检查结果**：是否有硬性阻断项
- **各维度评分**：具体扣分项和引用来源
- **待修复问题清单**：按 P0/P1 分级的详细修复指引

审核不通过后，按以下流程修复：
1. 读取审核报告，定位"待修复问题清单"
2. 逐条修复
3. 返回当前阶段重新执行质量审核
4. 重审报告命名为 `_quality_audit_{阶段}_v2.md`

> 审核报告的每个扣分项都有"引用来源"字段，指向产物中的具体章节或行号，
> 可以直接定位到问题位置。

## Q4：CONFIRM 阶段不小心关闭了对话怎么办？

**问题**：我在 CONFIRM 阶段还没确认就关闭了对话窗口，下次打开需要重新开始吗？

**解答**：
不需要重新开始。ReqPlan-v3 的接力棒机制支持跨 Session 续跑：

1. **重新激活**：再次触发 ReqPlan-v3 时，AI 会读取接力棒
2. **状态识别**：接力棒状态为 CONFIRM，AI 知道需要等待用户确认
3. **恢复摘要**：AI 会重新读取 `_analysis.md`，再次展示需求摘要
4. **继续确认**：用户直接确认即可进入 DESIGN 阶段

> 接力棒中保留了上次的分析摘要引用，AI 可以快速恢复上下文。
> 如果 `_analysis.md` 也被删除了，按 Q2 的恢复流程操作。

## Q5：如何跳过某些阶段（如文档完善流程）？

**问题**：我只需要补充项目文档，不需要完整的开发流程，能否跳过 DESIGN 或 IMPLEMENT 阶段？

**解答**：
可以。ReqPlan-v3 支持条件性跳过机制，适用于指定流程：

| 流程 | 可跳过的阶段 | 条件 |
|------|-------------|------|
| 文档完善流程 | DESIGN | 明确标记跳过 |
| 设计评审流程 | IMPLEMENT | 明确标记跳过 |
| 文档完善流程 | VERIFY | 无代码变更时跳过 |

**操作方式**：
1. 在 CONFIRM 阶段，用户描述需求时明确说明"只需要文档修改，不需要设计"
2. AI 会标记对应阶段为跳过
3. 被跳过的阶段产物流入标注"该阶段已跳过，原因：[跳过条件]"

**限制**：
- 同一阶段最多连续跳过 1 次
- 用户明确要求不跳过时 → 禁止跳过
- **首次运行（接力棒为 START）时禁止跳过任何阶段**
- **自绑定审查流程（流程8）不允许跳过任何阶段**

## Q6：多人/跨 Session 使用接力棒会冲突吗？

**问题**：我和同事轮流使用同一个项目，接力棒会不会互相覆盖？

**解答**：
ReqPlan-v3 的接力棒是单线程设计，不支持并发写入。使用建议：

1. **单人单任务原则**：接力棒追踪单个任务的执行状态
2. **任务完成后再启动新任务**：DONE 状态的接力棒会自动触发"新任务重置"机制
3. **协作流程**：
   - 同事 A 完成任务 → 接力棒 DONE
   - 同事 B 启动新任务 → 接力棒自动重置为 START
   - 历史产物保留在 `.agent/harness/` 中，可通过版本控制追溯

4. **如需并行工作**：推荐使用不同项目路径或分支

> 如果发现接力棒状态异常（如预期是 DONE 但显示 ANALYZE），可能是
> 上一个任务被中断。按"用户中断处理机制"操作：选择"立即重置"清理状态，
> 或"记入 TODO"待当前任务完成后自动处理。

## Q7：阶段执行中途任务被中断（如 AI 超时）怎么恢复？

**问题**：AI 在执行 IMPLEMENT 阶段时中断了（超时/网络断开），重新激活后能从中断点继续吗？

**解答**：
可以。按以下流程恢复：

1. **自动恢复**：重新激活 Skill 后，AI 自动读取接力棒
2. **阶段检测**：接力棒状态为 IMPLEMENT，AI 进入 IMPLEMENT 阶段
3. **进度识别**：AI 读取 `_implementation.md` 和接力棒中的"实施进度"
4. **从断点继续**：读取标记为"进行中"或"待开始"的任务继续执行

**手动恢复辅助**：
如果自动恢复不精确，可以：
1. 手动编辑接力棒，在"实施进度"表中标记已完成的任务
2. 然后重新激活 Skill

> 重要：如果中断时产物文件只写入了一半，AI 重新执行时会覆盖不完整的产物。
> 所有产物修改遵循原子性写入原则。

## Q8：如何为不支持的编程语言配置验证？

**问题**：我的项目使用 Rust 语言，但验证方案只显示了 Python（pylint/pytest）的示例，怎么办？

**解答**：
验证方案模板中的命令是示例性的，实际执行时 AI 会根据项目技术栈自动适配：

1. **项目检测**：ANALYZE 阶段 AI 会识别项目的技术栈
2. **验证适配**：AI 根据识别的技术栈自动选择对应命令
3. **常见技术栈映射**：

| 语言/框架 | Layer 1 静态检查 | Layer 2 测试 | Layer 3 构建 |
|-----------|-----------------|-------------|-------------|
| Python | ruff/pylint/mypy | pytest | python -m py_compile |
| JavaScript/TypeScript | eslint/prettier | jest/vitest | npm run build |
| Java | checkstyle/pmd | junit | mvn compile |
| Go | go vet/staticcheck | go test | go build |
| Rust | clippy | cargo test | cargo build |
| C# | dotnet format | dotnet test | dotnet build |

4. **自定义验证**：如果 AI 未自动识别，可在 CONFIRM 阶段补充说明技术栈：
   ```
   "请确认：项目使用 Rust 语言，验证命令使用 cargo clippy / cargo test / cargo build"
   ```

## Q9：产物文件被误修改/覆盖了怎么回滚？

**问题**：我手动编辑了 `_design.md`，但改错了，能找回原始版本吗？

**解答**：
回滚方案取决于是否启用版本控制：

### 方案1：已启用 Git 版本控制（推荐）
```bash
# 查看产物文件的变更历史
git log .agent/harness/_design.md

# 回滚到指定版本
git checkout <commit-hash> -- .agent/harness/_design.md
```

### 方案2：未启用版本控制
- 检查 `.agent/harness/` 目录下是否有命名备份（如 `_design.md.bak`）
- 如果产物是新生成的，重新执行当前阶段即可（AI 会覆盖产物文件）
- 如果产物涉及多个阶段且不想丢失进度：

  > 建议：至少为 `.agent/harness/` 目录启用 Git 版本控制。
  > ReqPlan-v3 的长期归档位于 `docs/harness/history.yaml` 和 `docs/harness/decisions.yaml`。

### 防止误修改的建议
- 产物文件由 AI 自动生成，不建议手动编辑
- 如需修改需求，在 CONFIRM 阶段选择"修改需求"，让 AI 重新生成
- 归档版本的产物文件不可变，用于审计和追溯

## Q10：流程执行到一半发现需求理解有偏差怎么办？

**问题**：我已经执行到 IMPLEMENT 阶段了，但发现之前分析的需求理解有偏差，怎么办？

**解答**：
ReqPlan-v3 的用户中断处理机制专门处理这种情况：

1. AI 暂停当前操作，展示 3 个选项：
   ```
   ⚠️ 我注意到您在流程执行中提出了新的需求/问题。
   请选择处理方式：
   
   ① 立即重置：中断当前流程，回到 ANALYZE 阶段重新分析并包含新需求
   ② 记入 TODO：将新需求记入接力棒"待办清单"，当前流程完成后自动发起新任务
   ③ 仅讨论：继续当前任务，暂不调整或新增
   ```

2. **推荐选择 ① 立即重置**：
   - 接力棒状态更新为 ANALYZE（保留已有产物）
   - 记录中断原因和修正后的需求
   - 进入 ANALYZE 阶段重新分析

3. **如果偏差不大，可选 ② 记入 TODO**：
   - 在接力棒新增"待办清单"章节记录新需求
   - 当前流程继续执行到 DONE
   - DONE 后自动触发"新任务重置"规则，启动修正任务

4. **如果只是讨论，可选 ③ 仅讨论**：
   - AI 回答用户问题后继续当前阶段
   - 不修改任务范围

> 不需要从头开始。已完成的产物保留在 `.agent/harness/` 中，
> 重新 ANALYZE 时可以复用已有分析结果。
